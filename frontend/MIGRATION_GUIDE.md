# Migration Guide: Frontend Configuration Parameterization

This guide walks you through migrating from hardcoded configuration values to the new parameterized configuration system.

## Overview

The new configuration system:
- Centralizes all configuration in `config.json`
- Uses environment variables for sensitive values
- Validates configuration at build time
- Maintains backward compatibility with fallbacks

## Migration Steps

### Step 1: Verify Current Setup

Before migrating, verify your current configuration values:

1. Check `frontend/src/authConfig.js` for:
   - `clientId`
   - `authority`
   - `redirectUri`
   - `scopes`

2. Check `frontend/src/config.js` for:
   - `AGENTCORE_ENDPOINT`
   - `WS_SIGN_ENDPOINT`

3. Check `frontend/src/app.js` for:
   - `maxAuthRetries`
   - Cookie expiration time (in `setCookie` function)

**Document these values** - you'll need them for the next step.

---

### Step 2: Set Up Environment Variables

1. Copy the environment template:
   ```bash
   cd frontend
   cp .env.example .env.local
   ```

2. Edit `.env.local` with your actual values:
   ```bash
   # Use the values you documented in Step 1
   MSAL_CLIENT_ID=your-client-id-here
   MSAL_AUTHORITY=https://login.microsoftonline.com/your-tenant-id
   REDIRECT_URI=https://your-domain.com/
   AGENTCORE_ENDPOINT=https://your-api.execute-api.region.amazonaws.com/invoke
   WS_SIGN_ENDPOINT=
   ```

3. Verify `.env.local` is in `.gitignore`:
   ```bash
   grep ".env.local" ../.gitignore
   ```

---

### Step 3: Install Dependencies

Install the required npm packages:

```bash
cd frontend
npm install
```

This installs:
- `ajv` - JSON schema validator used by the build script

---

### Step 4: Build Configuration

Run the configuration builder:

```bash
npm run build-config
```

**Expected output**:
```
Building configuration...
Reading configuration from /path/to/frontend/config.json
Substituting environment variables...
Reading schema from /path/to/frontend/config.schema.json
Validating configuration...
Generating configuration module...
Configuration written to /path/to/frontend/src/config.generated.js
Build complete!
```

**If you see errors**:
- Check that all environment variables are set correctly
- Verify values match expected formats (UUIDs, URLs)
- See "Troubleshooting" section below

---

### Step 5: Verify Generated Configuration

Check that `config.generated.js` was created:

```bash
ls -la src/config.generated.js
```

Review the generated file (optional):

```bash
cat src/config.generated.js
```

It should contain:
```javascript
// This file is auto-generated by build-config.js
// DO NOT EDIT MANUALLY

window.APP_CONFIG = {
  "auth": {
    "clientId": "your-actual-client-id",
    // ... other values
  },
  // ...
};
```

**Important**: This file should NOT be committed to version control.

---

### Step 6: Test Locally

1. Open `frontend/src/index.html` in a browser or serve it locally

2. Open browser console and check for:
   - No errors about missing `APP_CONFIG`
   - No warnings about using fallback values
   - Successful authentication flow

3. Verify configuration is loaded:
   ```javascript
   // In browser console
   console.log(window.APP_CONFIG);
   ```

**Expected behavior**:
- Application loads without errors
- Authentication works correctly
- API calls succeed
- No console warnings about missing configuration

---

### Step 7: Update CI/CD Pipeline

Add environment variables to your deployment pipeline:

**For AWS CodePipeline/CodeBuild**:
1. Go to CodeBuild project settings
2. Add environment variables:
   - `MSAL_CLIENT_ID`
   - `MSAL_AUTHORITY`
   - `REDIRECT_URI`
   - `AGENTCORE_ENDPOINT`
   - `WS_SIGN_ENDPOINT`

**For GitHub Actions**:
```yaml
env:
  MSAL_CLIENT_ID: ${{ secrets.MSAL_CLIENT_ID }}
  MSAL_AUTHORITY: ${{ secrets.MSAL_AUTHORITY }}
  REDIRECT_URI: ${{ secrets.REDIRECT_URI }}
  AGENTCORE_ENDPOINT: ${{ secrets.AGENTCORE_ENDPOINT }}
  WS_SIGN_ENDPOINT: ${{ secrets.WS_SIGN_ENDPOINT }}
```

**For other CI/CD systems**:
- Consult your platform's documentation for setting environment variables
- Ensure variables are available during the build step

---

### Step 8: Test Deployment

1. Deploy to a test/staging environment first:
   ```bash
   cd ../ai-agent/src/utils
   ./deploy-ui.sh
   ```

2. Verify the deployment:
   - Check that build configuration step succeeds
   - Verify `config.generated.js` is uploaded to S3
   - Test the deployed application
   - Check browser console for errors

3. If successful, deploy to production

---

### Step 9: Clean Up (Optional)

Once you've verified the new system works, you can optionally remove the hardcoded fallback values from `configAccessor.js`. However, keeping them provides an extra safety net.

**To remove fallbacks**:
1. Edit `frontend/src/configAccessor.js`
2. Replace fallback return statements with error throws:
   ```javascript
   getAuthConfig() {
     const config = window.APP_CONFIG?.auth;
     if (!config) {
       throw new Error('APP_CONFIG.auth not found - configuration not loaded');
     }
     return config;
   }
   ```

**Recommendation**: Keep fallbacks during initial rollout for safety.

## Troubleshooting

### Build Error: "Environment variable X is not defined"

**Problem**: Required environment variable is missing.

**Solution**:
1. Check `.env.local` exists and contains the variable
2. Verify variable name matches exactly (case-sensitive)
3. If using bash, source the file: `source .env.local`
4. For CI/CD, verify variable is set in pipeline configuration

### Build Error: "Configuration validation failed"

**Problem**: Configuration values don't match expected formats.

**Solution**:
1. Check error message for specific field that failed
2. Verify format requirements in `CONFIG_REFERENCE.md`
3. Common issues:
   - Client ID not in UUID format
   - URLs missing `https://` prefix
   - Numeric values out of range

### Application Uses Hardcoded Values

**Problem**: Application falls back to hardcoded defaults.

**Solution**:
1. Check browser console for warnings
2. Verify `config.generated.js` exists in `src/`
3. Check `index.html` loads `config.generated.js` before other scripts
4. Clear browser cache and reload

### Deployment Fails

**Problem**: Deployment script fails during configuration build.

**Solution**:
1. Run `npm run build-config` manually to see detailed errors
2. Check that all environment variables are set in deployment environment
3. Verify `config.generated.js` is created successfully
4. Check deployment logs for specific error messages

### Authentication Fails After Migration

**Problem**: Users can't sign in after deploying new configuration.

**Solution**:
1. Verify `redirectUri` matches exactly what's registered in Azure AD
2. Check that `clientId` and `authority` are correct
3. Verify `scopes` are properly formatted
4. Check browser console for MSAL errors
5. Test with a fresh browser session (clear cookies/cache)

### API Calls Fail

**Problem**: Backend API requests fail after migration.

**Solution**:
1. Verify `agentcoreEndpoint` is correct and accessible
2. Check browser network tab for actual endpoint being called
3. Verify endpoint URL format (should include protocol)
4. Test endpoint directly with curl or Postman

## Rollback Procedure

If you need to rollback to the old system:

### Quick Rollback (Fallback Values)

The application automatically falls back to hardcoded values if `APP_CONFIG` is not available. To trigger this:

1. Remove or rename `config.generated.js`:
   ```bash
   mv src/config.generated.js src/config.generated.js.bak
   ```

2. Redeploy:
   ```bash
   cd ../ai-agent/src/utils
   ./deploy-ui.sh
   ```

The application will use hardcoded fallback values.

### Full Rollback (Revert Changes)

To completely revert to the old system:

1. Revert code changes:
   ```bash
   git revert <commit-hash>
   ```

2. Remove new files:
   ```bash
   rm config.json config.schema.json build-config.js package.json
   rm src/configAccessor.js src/config.generated.js
   rm README.md CONFIG_REFERENCE.md MIGRATION_GUIDE.md
   ```

3. Restore original files from git history

4. Redeploy

## Best Practices

### Security

1. **Never commit sensitive values**:
   - Keep `.env.local` in `.gitignore`
   - Use environment variables for all sensitive data
   - Rotate credentials regularly

2. **Use different credentials per environment**:
   - Separate client IDs for dev/staging/prod
   - Different API endpoints per environment
   - Isolated Azure AD tenants if possible

3. **Validate in CI/CD**:
   - Ensure all required variables are set
   - Fail build if configuration is invalid
   - Test configuration in staging before production

### Maintenance

1. **Document configuration changes**:
   - Update `.env.example` when adding new variables
   - Update `CONFIG_REFERENCE.md` with new parameters
   - Communicate changes to team

2. **Version control**:
   - Commit `config.json` with placeholders
   - Never commit `.env.local` or `config.generated.js`
   - Track schema changes in `config.schema.json`

3. **Testing**:
   - Test configuration changes in staging first
   - Verify all environments after updates
   - Keep fallback values during initial rollout
